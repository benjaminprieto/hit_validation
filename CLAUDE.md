# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What This Is

A modular pipeline (Python) for **rigorous validation of virtual screening hits** using DOCK6 with footprint-based interaction analysis and GB/SA implicit solvation. Evaluates each hit on its own absolute merits to produce evidence for purchase decisions and pharmacophore template selection. Optionally imports pre-computed reference data as informational context in output tables (configurable via `reference_context` in campaign config). Written in Spanish (README, comments) by Benjamin Prieto / Environ Bio.

### The Iterative Drug Discovery Cycle

```
reference_docking -> molecular_docking -> HIT_VALIDATION -> reference_docking [iteration 2] -> ...
```

- **`reference_docking`**: Establishes the crystallographic reference baseline + generates pharmacophore
- **`molecular_docking`**: High-throughput virtual screening against molecule libraries
- **`hit_validation`** (THIS PROJECT): Rigorous validation of screening hits — per-molecule evidence for purchase or template selection. The winning molecule's mol2 feeds back into `reference_docking` iteration 2.

### Fundamental Differences vs reference_docking

| Aspect | reference_docking | hit_validation |
|--------|-------------------|----------------|
| Subject | Single crystallographic ligand | Multiple screening hits (10-20 molecules) |
| Coordinates | Crystal coordinates from PDB HETATM | Generated by docking (no crystal pose) |
| Ligand prep | No antechamber (preserves crystal geometry) | Antechamber required (AM1-BCC + Sybyl) |
| Box margin | 1.0 A (tight, keeps ligand near crystal pose) | 5.0 A (exploration space for hits) |
| Docking method | Rigid (crystal pose stays put) | Flex (anchor-and-grow conformational search) |
| PLIP input | Crystal complex (receptor + co-crystallized ligand) | Docked complex (receptor + best docked pose) |
| GB/SA minimize | false (crystal geometry is sacred) | true (docked poses benefit from relaxation) |
| Reference ligand | IS the subject | Optional context (configurable via `reference_context`) |
| Output | Reference energy profile | Per-molecule validation + selected pose mol2 export |
| Pharmacophore | 06a/06b generate pharmacophore | No pharmacophore — winning mol2 goes to reference_docking |

### Module Numbering = Execution Order

Pipeline: `00a -> 00b -> 00d -> 01b -> 01c -> 01d -> 01f -> 01e -> 03a -> 04b -> 01g -> 01h -> 01i -> [consolidation, only when N>1] -> 06a -> 06b -> 07a -> 07b -> 07c -> 07d`

When `n_replicas > 1`, modules 01c through 01i run N times under `replica_*/`,
then per-module `--consolidate-replicas` produces `<module>/consolidated/` and
`_replica_metadata/representative_per_mol.csv`. Post-replica modules
(06a, 06b, 07a, 07b, 07c, 07d) read from `<module>/consolidated/`.

## Configuration: `n_replicas`

Each campaign's number of replicas is determined by precedence:

1. **`campaign_config.yaml`** — most specific (recommended for new campaigns).
2. **`03_configs/pipeline_config.yaml`** — fallback global.
3. **Default** — `1` if neither defines it.

Recommended: define `n_replicas` explicitly in each campaign_config to ensure traceability:

```yaml
# 04_data/campaigns/<campaign_id>/campaign_config.yaml
campaign_id: "..."
# ... other fields ...
n_replicas: 3   # 1 for legacy single-run, N>1 for replicated UQ
```

The `pipeline_config.yaml` retains `n_replicas` as a fallback for backward
compatibility, but new campaigns should define it per-campaign.

`run_pipeline.sh` echoes `[CONFIG] n_replicas = <N> (source: <campaign_config|pipeline_config (fallback)>)`
at startup so the chosen value and its source are auditable in the run log.

Helpers under `02_scripts/utils/`:

- `audit_campaign_ids.py` — verifies that the internal `campaign_id` of every
  `campaign_config.yaml` matches its directory name. Run before relaunching a
  campaign to catch copy-paste bugs.
- `migrate_n_replicas_to_campaign_configs.py` — idempotent; appends
  `n_replicas: <default>` to any campaign_config that does not have it yet.

## Module-by-Module Specification

### 00a -- Ligand Preparation

Prepares N screening hit ligands using antechamber with AM1-BCC charges + Sybyl atom types.

**3-tier fallback:**
1. `antechamber -c bcc -at sybyl` (full AM1-BCC + Sybyl)
2. `antechamber -c gas -at sybyl` (Gasteiger + Sybyl, fast, no QM)
3. `obabel` conversion (last resort, warns about degraded atom typing)

**Config:** `atom_type: "sybyl"` (MUST be sybyl), `charge_method: "bcc"`, `timeout_per_molecule: 300`

### 00b -- Receptor Preparation

ChimeraX DockPrep -> `rec_charged.mol2` with AMBER ff14SB charges and Sybyl atom types.

### 00d -- Binding Site Definition (skipped if precomputed grids)

Trims receptor PDB for grid generation. Skipped if `precomputed_grids_dir` is set.

Default method: `reference_ligand` — uses a crystallographic ligand mol2 (`reference/UDX.mol2`) to define the binding pocket center. Alternative: `residues` (not recommended, manual and imprecise).

### 01b -- Grid Generation (skipped if precomputed grids)

DMS surface -> sphere generation -> sphere selection (near reference ligand) -> box (5.0 A margin) -> energy grids.

### 01c -- DOCK6 Docking

DOCK6 flex docking with grid_score primary. `pruning_clustering_cutoff: 2.0` (critical).

### 01d -- Footprint Rescore (self-reference mode)

Per-residue vdW + ES decomposition. Each hit uses its own scored pose as reference (self-reference).

### 01f -- GB/SA Hawkins Rescore

Implicit solvation scoring. `minimize: true` for docked poses.

### 01e -- Score Collection

Parses all DOCK6 scored mol2 files. `keep_all_poses: true` generates `dock6_all_poses.csv` for 07a.

### 03a -- PLIP Interaction Analysis

Runs PLIP on docked poses. Batch processes all molecules.

### 04b -- Footprint Analysis (dynamic zones + optional reference)

Per-residue energy decomposition with residue mapping (mol2 sequential -> PDB original numbering).

**Zones come from `campaign_config.yaml`** — no hardcoded zone definitions. If `reference_context.footprint_csv` is provided, reference data is imported as context row.

**Core function:** `run_footprint_analysis(..., zones=None, reference_context_csv=None, reference_context_label="Reference")`

**Output:**
- `footprint_per_molecule.csv` -- residue x molecule energy matrix (PDB numbering)
- `residue_consensus.csv` -- which residues always contribute
- `hit_vs_reference_comparison.csv` -- per-residue delta vs reference (if reference provided)
- `subpocket_coverage.csv` -- zone coverage per molecule (+ reference row if available)
- `ranking_by_zone.csv` -- hits ranked by energy per zone
- `pharmacophore_residues.json` -- residues contacted by >80% of molecules
- `molecule_footprint_summary.csv` -- one row per molecule with totals
- `binding_site_zones.html` -- interactive zones report

### 07a -- Decision Report (pose selection + mol2 export)

Multi-criteria pose selection + consolidated HTML report + CSV ranking.

**Pose selection:** For each molecule, selects the best pose from ~100 docked poses using weighted composite scoring (Grid Score, GB/SA, footprint vdW/ES). Weights configurable in `03_configs/07a_decision_report.yaml`.

**Classification logic (absolute thresholds):**
- **"Strong Pharmit template candidate"**: covers >= 2 zones with meaningful energy (< -0.5 kcal/mol per zone)
- **"Purchase candidate"**: Grid Score <= -30.0 AND covers >= 1 zone
- **"Weak candidate"**: otherwise

Zone definitions and reference context come from `campaign_config.yaml` — nothing hardcoded.

**Output:**
- `decision_report.html` -- one card per molecule with pose selection, scores, zones, PLIP
- `decision_summary.csv` -- machine-readable ranking
- `pose_selection_summary.csv` -- per-molecule pose selection details
- `selected_poses/{name}.mol2` -- best pose mol2 (feeds reference_docking iteration 2)

## DOCK6.13 Scoring Architecture

DOCK6.13 silently ignores all `_secondary` scoring functions during flexible docking. Therefore:
- **01c** uses only `grid_score_primary` (docking step)
- **01d** uses `footprint_similarity_score_primary` (separate rescore)
- **01f** uses `gbsa_hawkins_score_primary` (separate rescore)

Each scoring function runs as an independent rigid rescore pass with `orient_ligand=no`.

**DOCK6 template strings** in dock6_runner.py, footprint_rescore.py, gbsa_rescore.py, footprint_rescoring.py contain **zero** `_secondary` parameters. This is verified by tests.

## Two-Layer Design

Each module has exactly two files:
1. **Core** (`01_src/hit_validation/module.py`): Pure logic, receives paths/dicts, returns results.
2. **Script** (`02_scripts/module.py`): Thin CLI wrapper that merges YAML config + campaign_config + CLI overrides.

## Campaign Config is the Source of Truth

`campaign_config.yaml` defines everything: receptor, ligands, pH, binding site, zones, reference data. Module YAMLs control only algorithm parameters.

Key campaign config sections:
- **`grids.binding_site`**: Default `method: "reference_ligand"` with `reference_mol2` pointing to a crystallographic ligand mol2 in `reference/`. This defines the 3D geometry of the binding pocket for sphere generation and box placement. NOT used for scoring.
- **`zones`**: Binding site zone definitions (zone_id -> {label, residues, druglike}). Used by 04b, 01h, 07a.
- **`reference_context`**: Optional pre-computed reference data for comparison (label, scores_csv, plip_json, footprint_csv). Works with any reference ligand.

**Important distinction**: `reference_mol2` (binding site geometry) and `reference_context` (pre-computed scores for comparison) are independent. Both may point to the same ligand (e.g., UDX), but serve completely different purposes.

## Key Domain Constraints

- **DOCK6 80-char path limit**: Fortran programs truncate paths. The pipeline uses symlinks.
- **atom_type MUST be sybyl**: GAFF2 causes DOCK6 to silently fall back to rigid docking.
- **pruning_clustering_cutoff MUST be 2.0**: DOCK6 default of 100 A merges all poses into 1 cluster.
- **Receptor mol2 must have AMBER ff14SB charges** and Sybyl atom types.
- **Binding site** defined by reference ligand mol2 (default) or residue list (alternative). Use `precomputed_grids_dir` to skip grid generation.
- **PLIP protonates internally** -- do NOT pre-protonate ligand poses before PLIP.
- **Zones are dynamic**: All zone definitions come from `campaign_config.yaml`. No hardcoded zones in code.

## Per-frame Analysis

The pipeline computes per-frame metrics from MD trajectories:

- **ΔG per frame** (01h): GB and PB energies extracted from MMPBSA sander outputs
  (`_MMPBSA_<species>_<gb|pb>.mdout.0`).
  Output: `<mol>_dG_per_frame.csv` per molecule with columns
  `frame, time_ns, dG_GB, vdW_GB, EEL_GB, EGB, ESURF, dG_PB, vdW_PB, EEL_PB, EPB, nonpolar_PB`.
- **Distances per frame** (01i): `d_min` (atom-atom minimum) and `d_func` (to chemically
  relevant atoms, see `FUNCTIONAL_ATOMS`) for residues in `functional_residues` of campaign_config.
  Output: `distances_per_frame.csv` per molecule.

Both modules produce mean ± std consolidations across N replicas via
`consolidate_dG_per_frame_replicas()` (01h) and
`consolidate_distances_per_frame_replicas()` (01i).

### Config

`03_configs/01g_mmpbsa_decomp.yaml`:
- `production_ns: 50.0` — MD production length.
- `mmpbsa_interval_ps: 1000.0` — 1 MMPBSA point per ns (50 points total).
- `mmpbsa.istrng/fillratio/radiopt` — PB parameters (PB runs alongside GB).

### Outputs

Per replica:
- `replica_N/01h_mmpbsa_analysis/<mol>_dG_per_frame.csv`
- `replica_N/01i_trajectory_analysis/<mol>/distances_per_frame.csv`

Consolidated (N≥2 replicas):
- `01h_mmpbsa_analysis/<mol>_dG_per_frame_consolidated.csv` (mean ± std × replicas)
- `01i_trajectory_analysis/<mol>/distances_per_frame_consolidated.csv`
